## 1. Анализ текущей экономики бота

### 1.1. Динамика роста

Бот демонстрирует экспоненциальный рост по всем ключевым метрикам:

```
MAU (месяцы):         Расходы OpenAI (₽):

171 |          *       8262 |          *
    |                       |
    |                       |
102 |     *                 |
    |                  2557 |     *
 42 | *                     |
    +---------->        559 | *
    Дек  Янв  Фев          +---------->
                            Дек  Янв  Фев
```

**Темпы роста** (месяц к месяцу):

| Метрика          | Дек -> Янв | Янв -> Фев | Средний MoM |
|-----------------|-----------|-----------|-------------|
| MAU              | +143%     | +68%      | +102%       |
| Транскрипций     | +214%     | +94%      | +147%       |
| Аудио минут      | +357%     | +223%     | +288%       |
| Расходы OpenAI   | +357%     | +223%     | +288%       |

**Критическое наблюдение**: расходы растут быстрее, чем количество пользователей. Причина -- новые пользователи начинают транскрибировать все более длинные файлы. Средняя длительность аудио на транскрипцию выросла:

| Месяц   | Ср. длительность файла | Ср. расход на пользователя |
|---------|----------------------|--------------------------|
| Дек 2025 | 3.2 мин              | 13.3 ₽                   |
| Янв 2026 | 4.7 мин              | 25.1 ₽                   |
| Фев 2026 | 7.8 мин              | 48.3 ₽                   |

### 1.2. Структура себестоимости (февраль 2026)

```
Структура расходов: 9 031 ₽/мес
+--------------------------------------------------+
|                                                  |
|            OpenAI Whisper API                    |
|               8 262 ₽                            |
|              (91.5%)                             |
|                                                  |
+--------------------------------------------------+
|  VPS 672₽ (7.4%)  |  DeepSeek 97₽ (1.1%)       |
+--------------------------------------------------+
```

| Компонент          | Расход (₽/мес) | Доля    | Тип расхода   |
|-------------------|---------------|---------|--------------|
| OpenAI Whisper API | 8 262          | 91.5%   | Переменный   |
| Хостинг VPS        | 672            | 7.4%    | Постоянный   |
| DeepSeek API       | 97             | 1.1%    | Переменный   |
| **Итого**          | **9 031**      | **100%** |              |

**Ключевой cost driver**: OpenAI Whisper API -- это 91.5% всех расходов. Стоимость линейно пропорциональна количеству минут аудио.

**Себестоимость единицы продукта**:
- 1 минута аудио = 0.67 ₽ (Whisper API)
- 1 обработка текста (DeepSeek) = ~0.04 ₽
- **Полная себестоимость 1 транскрипции** = 0.67 * средн. длительность + 0.04 = **5.26 ₽** (при ср. 7.8 мин)

### 1.3. Ключевой паттерн затрат: правило длинных файлов

```
Распределение расходов по длительности файлов:

                                     % расходов
0-30с   [=]                            1.0%
31-60с  [==]                           2.3%
1-2м    [===]                          4.0%
2-5м    [======]                       7.6%
5-10м   [=====]                        5.8%
10-30м  [=============]               15.9%
30-60м  [====================]        23.4%
60+м    [================================]  39.9%
        +----+----+----+----+----+
        0%   10%  20%  30%  40%
```

**12% файлов (30+ минут) генерируют 63.3% расходов** -- это главный рычаг для монетизации. Стратегия ограничений должна фокусироваться именно на длинных файлах.

| Мар 2026 | 205                      | 188                 | 171             |
| Апр 2026 | 246                      | 207                 | 171             |
| Май 2026 | 295                      | 228                 | 171             |
| Июн 2026 | 354                      | 250                 | 171             |
| Июл 2026 | 425                      | 275                 | 171             |
| Авг 2026 | 510                      | 303                 | 171             |

**Прогноз расходов БЕЗ монетизации** (при базовом росте +10% MoM):

| Месяц    | MAU | Аудио мин (прогноз) | Расход OpenAI (₽) | Полный расход (₽) |
|---------|-----|---------------------|-------------------|--------------------|
| Фев 2026 | 171 | 12 331              | 8 262             | 9 031              |
| Мар 2026 | 188 | 13 564              | 9 088             | 9 857              |

### 2.3. Влияние различных порогов бесплатного лимита

Моделирование: какой % пользователей и расходов затронет каждый порог.

**Порог: дневной лимит бесплатных минут**

| Бесплатный лимит | % юзеров за лимитом | % расходов за лимитом | Экономия (₽/мес) | Потенциал монетизации |
|-----------------|--------------------|--------------------|------------------|---------------------|
| 5 мин/день       | ~35%               | ~85%               | ~7 023           | Агрессивный, высокий churn |
| 10 мин/день      | ~22%               | ~75%               | ~6 197           | Умеренный, хороший баланс |
| 20 мин/день      | ~14%               | ~60%               | ~4 472           | Мягкий, минимальный churn |
| 30 мин/день      | ~10%               | ~48%               | ~3 581           | Очень мягкий |
| Без лимита дня, лимит на файл 10 мин | ~18% | ~63% | ~5 245 | Таргетированный |

**Расчёт "порога боли"**: медианный пользователь (P50) использует 8.4 мин/мес = 0.28 мин/день. Даже 5 мин/день не затронет 65% пользователей. Но для высокосегментных пользователей (P95+) лимит 10 мин/день -- это существенное ограничение.

### 2.4. Кто потенциально готов платить

**Профиль платящего пользователя** (на основе данных):

1. **Power Users** (2 чел., расход 1 113 ₽) -- гарантированно заплатят, бот интегрирован в workflow.
   - tablian: 1 130 транскрипций, 57 дней активности -- это ежедневный рабочий инструмент
   - Fudjik: 59 транскрипций, 32 дня -- регулярное использование

2. **Heavy Filers** (расход 100+ ₽/мес, ~25 чел.) -- транскрибируют длинные записи (лекции, встречи, подкасты). Высокая WTP, т.к. альтернатива -- ручная расшифровка.

3. **Regular Users** с длинными файлами (70 чел. в Medium) -- 8-12% готовы платить при правильном ценообразовании.

**Оценка конвертируемой базы**: из 283 всех пользователей, реалистично конвертируемая аудитория -- 25-95 человек (9-34%), при прогнозных 274 MAU в марте это 25-93 платящих.

## 4. Варианты модели монетизации

### Вариант A: Freemium с дневным лимитом + подписка

**Описание**: бесплатный тариф с лимитом минут в день, платная подписка снимает ограничения.

**Параметры**:

| Параметр | Free | Pro (подписка) |
|----------|------|---------------|
| Лимит минут | 10 мин/день | Без лимита |
| Макс. длительность файла | 10 мин | Без ограничений |
| Цена | 0 ₽ | 199 ₽/мес |

**Расчёт потенциального дохода** (на базе текущих 171 MAU):

Расходы: Free-юзеры генерируют только короткие файлы (≤10 мин/день), Paid-юзеры — все файлы. Free-расход ~5₽/чел/мес, Paid-расход ~76₽/чел/мес (факт из данных). Постоянные расходы: 769₽ (VPS + DeepSeek). Доход по ЮKassa (комиссия 3.5%).

| CR | Платящих | Free | Доход (₽/мес) | Расходы (₽/мес) | Маржа (₽/мес) |
|----|---------|------|--------------|----------------|--------------|
| 5% | 9       | 162  | 1 728        | 2 263          | -535         |
| 10%| 17      | 154  | 3 265        | 2 831          | **+434**     |
| 15%| 26      | 145  | 4 993        | 3 470          | **+1 523**   |

Формула: Доход = Paid * 199 * 0.965. Расходы = Free * 5 + Paid * 76 + 769.

**При прогнозных 188 MAU (март, +10% MoM)**:

| CR  | Платящих | Free | Доход (₽/мес) | Расходы (₽/мес) | Маржа |
|-----|---------|------|--------------|----------------|-------|
| 5%  | 9       | 179  | 1 728        | 2 348          | -620  |
| 10% | 19      | 169  | 3 649        | 3 058          | **+591** |
| 15% | 28      | 160  | 5 377        | 3 697          | **+1 680** |

**Break-even**: ~8% CR при 171 MAU, ~7% CR при 250+ MAU.

**Плюсы**:
- Простая и понятная модель (как у Voice Messages)
- Предсказуемый ежемесячный доход
- Бесплатный тариф сохраняет вирусный рост
- 10 мин/день покрывает 78% пользователей -- минимальный friction

**Минусы**:
- Дневной лимит 10 мин не затрагивает большинство юзеров -- низкий стимул покупать
- Фиксированная подписка: power user на 1000 мин/мес платит столько же, сколько user на 50 мин/мес
- Break-even достигается только при 15% CR -- достаточно амбициозно
- Необходимость биллинг-инфраструктуры (Telegram Stars / ЮKassa / Stripe)

### Вариант B: Freemium с лимитом длительности файла + пакеты минут

**Описание**: бесплатная транскрипция коротких файлов (до 5 минут). Для длинных файлов -- покупка пакетов минут.

**Параметры**:

| Параметр | Free | Пакеты минут |
|----------|------|-------------|
| Макс. длительность файла | 5 мин | Без ограничений |
| Кол-во транскрипций/день | Без лимита | Без лимита |
| Минуты за лимитом | Не доступны | Списываются из пакета |

**Пакеты минут**:

| Пакет | Цена (₽) | Цена/мин (₽) | Скидка vs базовая |
|-------|----------|-------------|------------------|
| 30 мин | 59       | 1.97        | -                |
| 100 мин | 169      | 1.69        | -14%             |
| 300 мин | 449      | 1.50        | -24%             |
| 1000 мин | 1 290    | 1.29        | -35%             |

**Расчёт потенциального дохода**:

Файлы >5 мин = 22% всех файлов, но 88% расходов. Из текущих 283 активных юзеров, 145 отправляли хотя бы один файл >5 мин.

**Ключевой момент**: Free-юзеры ограничены 5 мин → они НЕ генерируют дорогие расходы. Средний расход на Free = 3.4₽/мес (только короткие файлы). Платящие: ~76₽/мес (все файлы).

Предполагаем средний чек 150₽/мес (пакет 100 мин, не все тратят полностью). Доход по ЮKassa (3.5%).

| CR  | Платящих | Free | Доход (₽/мес) | Расходы (₽/мес) | Маржа |
|-----|---------|------|--------------|----------------|-------|
| 5%  | 9       | 162  | 1 303        | 2 004          | -701  |
| 10% | 17      | 154  | 2 461        | 2 585          | -124  |
| 15% | 26      | 145  | 3 764        | 3 238          | **+526** |
| 20% | 34      | 137  | 4 922        | 3 819          | **+1 103** |

Формула: Доход = Paid * 150 * 0.965. Расходы = Free * 3.4 + Paid * 76 + 769.

**При прогнозных 188 MAU (март, +10%)**:

| CR  | Платящих | Free | Доход (₽/мес) | Расходы (₽/мес) | Маржа |
|-----|---------|------|--------------|----------------|-------|
| 5%  | 9       | 179  | 1 303        | 2 062          | -759  |
| 10% | 19      | 169  | 2 750        | 2 788          | -37   |
| 15% | 28      | 160  | 4 053        | 3 441          | **+612** |
| 20% | 38      | 150  | 5 500        | 4 167          | **+1 334** |

**Break-even**: ~10% CR при 188 MAU. Модель хуже Варианта A из-за более низкого ARPU (150₽ vs 199₽).

**Плюсы**:
- Таргетирует главный cost driver (длинные файлы = 63% расходов)
- Короткие файлы остаются бесплатными -- сохраняет вирусность
- Гибкость: пользователь покупает столько, сколько нужно
- Пакеты бессрочные -- ниже психологический барьер
- Хорошо работает для use case "лекции/встречи" -- длинные файлы = очевидная ценность

**Минусы**:
- Менее предсказуемый доход (vs подписка)
- Юзеры могут нарезать длинные файлы на куски по 5 мин (обход)
- Средний чек может быть ниже ожидаемого
- Сложнее в коммуникации, чем простая подписка

### ~~Вариант C: Гибридная модель~~ (отклонён)

Ранее рассматривалась гибридная модель с подпиской 149₽/мес (300 мин/мес) + пакеты. **Отклонена** по следующим причинам:

1. **Каннибализация**: подписка с месячными минутами (300 мин за 149-199₽) убивает продажу пакетов. Зачем покупать пакет 100 мин за 169₽, если подписка 199₽ даёт 300 мин?
2. **Сложность условий**: ограничение "5 мин/файл + 5 транскрипций/день" — слишком много условий для бесплатного периода
3. **Подписка с месячными минутами** не масштабируется в модель с дневным лимитом (как у конкурентов)

---

### Сводное сравнение вариантов

| Критерий | Вариант A (подписка) | Вариант B (пакеты) | **Выбранный (см. раздел 5)** |
|----------|---------------------|-------------------|--------------------------------|
| Break-even CR (171 MAU, ЮKassa) | ~8% | ~12% | ~8-10% |
| Предсказуемость дохода | Высокая | Низкая | **Высокая** (подписки + пакеты) |
| Сохранение вирусности | Средняя | Высокая | **Высокая** (10 мин/день бесплатно) |
| Покрытие сегментов | Один размер | Гибко | **Полное** |
| Конкурентность цены | На уровне рынка | Ниже рынка | **Ниже рынка** |
| Сложность внедрения | Низкая | Средняя | Средняя-высокая |
| Защита от cost-оверрана | **Высокая** | Высокая | **Высокая** |
| Потенциал LTV | Средний | Ниже | **Высокий** (подписки с 1 дня) |

## 5. Выбранная модель: Дневная квота + Подписки + Пакеты минут

### 5.1. Ключевая идея

Единый дневной лимит бесплатных минут для всех (10 мин/день). Сверх лимита — подписка или покупка пакетов. Подписочная модель запускается сразу с одним активным тарифом Pro (30 мин/день). Архитектура позволяет в будущем полностью убрать бесплатные минуты и оставить только подписки и пакеты.

**Почему именно эта модель?**

1. **Простота** — не нужно считать файлы, не нужно ограничивать их длительность. Одна метрика: минуты в день
2. **Нет каннибализации** — подписка даёт минуты **в день** (30 мин/день), а не суммарно в месяц. Пользователь, которому нужно транскрибировать одну лекцию на 2 часа, всё равно купит пакет
3. **Два канала дохода** — подписка (предсказуемый recurring) + пакеты (разовые покупки). Максимальное покрытие сегментов
4. **Конкурентное преимущество** — 10 мин/день бесплатно + AI-обработка текста + файлы до 2 ГБ
5. **Масштабируемость** — дневной лимит настраивается в БД (можно обнулить для перехода на полностью платную модель). Можно добавлять новые уровни подписок без изменения кода

### 5.2. Описание модели

#### 5.2.1. Бесплатный дневной лимит

Каждый пользователь получает **10 бесплатных минут в день**. Лимит обнуляется ежедневно в 00:00 MSK. Внутри дневного лимита — никаких ограничений: можно отправить одно 10-минутное аудио или десять по 1 минуте.

| Параметр | Значение | Примечание |
|----------|----------|-----------|
| Дневной лимит | 10 мин | Настраивается в БД, без деплоя |
| Ограничение длительности файла | Нет | Файл любой длины, если хватает квоты |
| Ограничение кол-ва транскрипций | Нет | Только суммарные минуты |
| Обнуление | Ежедневно в 00:00 MSK | |
| AI-кнопки | Без ограничений | Структурировать, Сделать красиво, О чём этот текст |

**Архитектурное требование**: значение лимита может быть установлено в **0** — это полностью отключает бесплатные минуты для всех пользователей, оставляя только подписки и пакеты.

**Почему 10 мин/день?**
- Медианный пользователь потребляет 0.28 мин/день — 10 мин покрывает **~78%** пользователей
- ~22% пользователей выходят за лимит, генерируя ~75% расходов — высокий потенциал монетизации
- Снижение расхода на Free-юзера: ~5₽/мес (vs ~9₽ при 20 мин/день) — модель выходит на безубыточность при более реалистичном CR
- Лимит можно будет увеличить позже как промо-акцию или снизить до 0 для полностью платной модели

#### 5.2.2. Подписочная модель

На старте один активный уровень подписки — **Pro** (30 мин/день):

| Параметр | Значение |
|----------|----------|
| Название | Pro (рабочее) |
| Дневной лимит | 30 мин/день |

**Периоды и ценообразование**:

| Период | Цена | Скидка от недельного |
|--------|------|----------------------|
| Неделя | 69 рублей разово | — |
| Месяц | 199 рублей разово (199 рублей/мес) | **-33%** |
| Год | 1668 рублей разово (139 рублей/мес) | **-53%** |

**Логика скидок**:
- Недельный тариф, пересчитанный на 30 дней = 296 рублей
- Месячный пакет даёт скидку **33%** от стоимости недельного, пересчитанного на месяц
- Годовой пакет даёт скидку **53%** от стоимости недельного, пересчитанного на месяц

**Архитектурные требования к подпискам**:
- Возможность создавать **произвольное количество уровней** подписки (1, 2, 4 и т.д.)
- Каждому уровню задаётся: название, дневной лимит минут, описание, порядок отображения
- Доступные периоды: **неделя, месяц, год** (в будущем возможен квартал)
- Для каждой комбинации (уровень × период) задаётся отдельная цена
- Подписка автоматически продлевается или истекает по дате окончания
- Пользователь может иметь только одну активную подписку

#### 5.2.3. Пакеты минут

Если аудио превышает оставшийся дневной лимит — пользователь покупает пакет минут. Пакеты **бессрочные** (не сгорают).

**Лестница тарифов** (настраивается в БД):

| Пакет | Цена (₽) | Цена/мин (₽) | Скидка vs базовая | Маржа (ЮKassa) | Позиционирование |
|-------|----------|---------------|-------------------|----------------|-----------------|
| 50 мин | 100 | 2.00 | — | 68% | Попробовать |
| 100 мин | 175 | 1.75 | -13% | 63% | Одна лекция |
| 500 мин | 750 | 1.50 | -25% | 57% | Популярный |
| 5 000 мин | 5 000 | 1.00 | -50% | 35% | Корпоративный |

**Маржа рассчитана** с учётом комиссии ЮKassa (3.5%):
- 50 мин: (100 × 0.965 − 50 × 0.67) / (100 × 0.965) = **68%**
- 100 мин: (175 × 0.965 − 100 × 0.67) / (175 × 0.965) = **60%**
- 500 мин: (750 × 0.965 − 500 × 0.67) / (750 × 0.965) = **54%**
- 5 000 мин: (5 000 × 0.965 − 5 000 × 0.67) / (5 000 × 0.965) = **31%**

> Примечание: при оплате через Telegram Stars (комиссия ~30%) маржа существенно ниже, особенно на крупных пакетах. Stars рекомендуются для мелких покупок (50 мин), ЮKassa — для крупных.

**Принципы ценообразования**:
- Первая ступень (50 мин / 100₽) — низкий абсолютный порог входа
- Каждая следующая ступень дешевле за минуту — пользователю **наглядно видна** выгода
- Все цены в коридоре рынка (1.00–2.00₽/мин vs рынок 1.50–3.33₽/мин)
- Пакет 5 000 мин (5 000₽) — для корпоративных клиентов и power users

**Гибкость**: система позволяет создавать пакеты на **любое** количество минут с любой ценой. Каждому пакету задаётся: название, кол-во минут, цена, порядок отображения.

#### 5.2.4. Welcome-бонус для новых пользователей

Каждый новый пользователь при первом запуске бота получает **60 бонусных минут** — для тестирования расшифровки длинных аудио (лекции, встречи, подкасты).

| Параметр | Значение | Настройка |
|----------|----------|-----------|
| Количество бонусных минут | 60 | Настраивается в БД |
| Срок действия | 7 дней / бессрочно | Настраивается в БД |
| Условие получения | Первый запуск бота | Автоматически |

**Варианты использования**:
- **Стандартный режим**: 60 минут на 7 дней — пользователь успевает попробовать длинные аудио и привыкнуть к сервису
- **Бессрочный режим**: 60 минут навсегда — для более мягкого привлечения
- **Акционный режим**: 120 минут на 14 дней — для рекламных кампаний (например, новогодняя акция)

Бонусные минуты тратятся **только** когда дневной лимит исчерпан. То есть в рамках 20 мин/день бонус не расходуется.

#### 5.2.5. Грейс-период для существующих пользователей

При введении монетизации все зарегистрированные пользователи получают **60 бонусных минут** (бессрочных) — через одноразовый миграционный скрипт. Минуты оформляются как бонусные без срока действия.

**Логика**:
- Если пользователь использует бот в рамках дневного лимита (10 мин/день) — бонусные минуты не тратятся, просто остаются на балансе
- Если пользователь отправит длинное аудио сверх дневного лимита — бонусные минуты списываются
- Это сохраняет лояльность пользователей, которые ещё не пробовали расшифровку длинных аудио

#### 5.2.6. Логика списания минут

```
Пользователь отправляет аудио (N минут)
           │
           ▼
┌─────────────────────────────┐
│  Остаток дневного лимита    │
│  (daily_remaining >= N?)    │
│  (бесплатные или из подписки)│
└──────────┬──────────────────┘
           │
     ДА    │    НЕТ
     ▼     │     ▼
┌──────────┐  ┌──────────────────────────────┐
│ Списать  │  │ daily_remaining +             │
│ из       │  │ bonus_balance +               │
│ дневного │  │ package_balance >= N?          │
│ лимита   │  └──────────┬───────────────────┘
└──────────┘       ДА    │     НЕТ
                   ▼     │      ▼
                ┌────────┐  ┌─────────────────────┐
                │Бесшовно│  │ Отказ + предложение  │
                │списать:│  │ купить пакет/подписку│
                │daily + │  │ с кнопками покупки   │
                │bonus + │  └─────────────────────┘
                │package │
                └────┬───┘
                     ▼
              ┌──────────────────┐
              │ Уведомление:     │
              │ «Дневной лимит   │
              │ закончился.      │
              │ Использовано X   │
              │ мин из пакета.   │
              │ Остаток: Y мин»  │
              └──────────────────┘
```

**Приоритет списания**:
1. Сначала — **дневной лимит** (бесплатные минуты — общие или из подписки)
2. Затем — **бонусные минуты** (welcome-бонус / грейс-бонус)
3. Последними — **купленные пакеты минут**

**Два сценария при отправке аудио**:

**А) Суммарно минут достаточно** (дневной остаток + бонус + пакет >= длительность аудио):
- Транскрипция выполняется **бесшовно**, без остановки
- После транскрипции — **обязательное уведомление**: «Дневной лимит закончился. Использовано X мин из пакета. Остаток: Y мин»

**Б) Суммарно минут недостаточно**:
- Транскрипция **не выполняется**
- Сообщение: «Недостаточно минут. Длительность: N мин. Доступно: X мин»
- Инлайн-кнопки покупки подходящих пакетов или подписки

```
⚠️ Недостаточно минут для расшифровки.

Длительность аудио: 45 мин
Доступно: 3 мин (дневной лимит) + 0 мин (пакет)

Докупите пакет минут:
[50 мин — 100₽]  [100 мин — 175₽]  [500 мин — 750₽]

Или оформите подписку Pro (30 мин/день): /subscribe
```

#### 5.2.7. Индивидуальные условия для пользователей

Возможность переопределить любой параметр для конкретного `user_id`:

| Параметр | Пример переопределения |
|----------|----------------------|
| Дневной лимит | Увеличить до 30/50 мин или сделать безлимитным |
| Пакетные предложения | Индивидуальная лестница цен |
| Welcome-бонус | Увеличенный или бессрочный бонус |
| Условия подписки | Индивидуальная цена или расширенный лимит |

**Приоритет**: индивидуальные условия > общие условия (оферта). Если индивидуальный параметр не задан (NULL) — берётся из общих условий.

#### 5.2.8. Сроки действия условий

Каждое условие (общее и индивидуальное) имеет:
- **valid_from** — дата начала действия
- **valid_to** — дата окончания (NULL = бессрочно)

Это позволяет: менять оферту с конкретной даты, создавать временные акции, планировать будущие изменения. Условия вступают в силу автоматически по дате.

#### 5.2.9. Платёжные провайдеры

| Провайдер | Комиссия | Особенности | Рекомендация |
|-----------|----------|-------------|--------------|
| **Telegram Stars** (Bot Payments API) | ~30% | Нативно в Telegram, zero friction | Мелкие покупки (пакет 50 мин) |
| **ЮKassa** (карты РФ, СБП) | 3.5% | Максимальная маржа | Основной канал, крупные пакеты и подписки |

**Влияние провайдера на маржу пакетов**:

| Пакет | Маржа (ЮKassa 3.5%) | Маржа (Stars 30%) |
|-------|---------------------|-------------------|
| 50 мин / 100₽ | 68% | 52% |
| 100 мин / 175₽ | 60% | 44% |
| 500 мин / 750₽ | 54% | 37% |
| 5 000 мин / 5 000₽ | 31% | 4% |

> **Важно**: пакет 5 000 мин через Telegram Stars практически не маржинален. Крупные покупки следует направлять в ЮKassa.

### 5.3. Архитектура системы тарифов: гибкая конфигурация в БД

Условия тарифов хранятся **в базе данных, а не в ENV-файле**. Это даёт возможность менять тарифы, вводить акции и индивидуальные условия без пересборки и деплоя.

#### Принцип: общие условия + индивидуальные переопределения

```
┌───────────────────────────────────────────────────────┐
│              Общие условия (оферта)                    │
│  Действует для ВСЕХ пользователей                      │
│  valid_from: 2026-03-15  valid_to: NULL (бессрочно)    │
│                                                        │
│  daily_free_minutes: 10                                │
│  subscription_tiers: [Pro: 30 мин/день]               │
│  welcome_bonus_minutes: 60                             │
│  welcome_bonus_days: 7                                 │
│  ...                                                   │
└───────────────────────────────────────────────────────┘
         ↓ переопределяется (если есть) ↓
┌───────────────────────────────────────────────────────┐
│        Индивидуальные условия (override)               │
│  Для КОНКРЕТНОГО пользователя                          │
│  valid_from: 2026-04-01  valid_to: 2026-04-30         │
│                                                        │
│  user_id: 12345                                        │
│  daily_free_minutes: 30  ← переопределено              │
│  welcome_bonus_minutes: NULL  ← не задано → берём      │
│                                из общих условий         │
└───────────────────────────────────────────────────────┘
```

#### Логика разрешения условий (приоритет)

Для каждого пользователя итоговые условия определяются по приоритету:

1. **Индивидуальные условия** (если есть активные на текущую дату для данного user_id) — высший приоритет
2. **Общие условия** (активная оферта на текущую дату) — базовый уровень

Если у индивидуального условия параметр не задан (NULL), берётся значение из общих условий. Это позволяет переопределять только нужные параметры, не дублируя остальные.

#### Управление сроками действия

Каждое условие (и общее, и индивидуальное) имеет:
- **valid_from** (дата начала) — с какого момента условие действует
- **valid_to** (дата окончания, nullable) — до какого момента; NULL = бессрочно

Это позволяет:
- **Менять общую оферту с конкретной даты**: создать новые общие условия с `valid_from = 2026-06-01` → старые условия действуют до этой даты, новые — после
- **Делать временные акции**: индивидуальные условия с `valid_from = 2026-04-01`, `valid_to = 2026-04-30` → акция на месяц
- **Давать бессрочные VIP-условия**: индивидуальные условия с `valid_to = NULL`
- **Планировать изменения заранее**: можно создать будущие условия, которые автоматически вступят в силу

#### Примеры использования

| Сценарий | Реализация |
|----------|-----------|
| Увеличить дневной лимит с 10 до 15 мин для всех с 1 июня | Новые общие условия: `daily_free_minutes=15`, `valid_from=2026-06-01` |
| Дать конкретному пользователю 50 мин/день на месяц | Индивид. условие: `user_id=X`, `daily_free_minutes=50`, `valid_from/to` |
| Дать конкретному пользователю безлимит навсегда | Индивид. условие: `user_id=X`, `daily_free_minutes=NULL (безлимит)`, `valid_to=NULL` |
| Акция «Новый год»: 120 бонусных минут на 14 дней | Общие условия: `welcome_bonus_minutes=120`, `welcome_bonus_days=14`, `valid_from=2026-12-20`, `valid_to=2027-01-10` |
| Индивидуальная лестница пакетов для VIP-юзера | Индивид. пакетные предложения: отдельные записи в `minute_packages` для user_id |
| Добавить уровень подписки Business (60 мин/день) | Новый tier в `subscription_tiers`: `business`, `daily_minutes=60`, цены по периодам |
| Полностью убрать бесплатные минуты | Общие условия: `daily_free_minutes=0` — только подписки и пакеты |

#### Параметры, управляемые через систему тарифов

| Параметр | Тип | Описание | Default (Free) |
|----------|-----|---------|----------------------|
| `daily_free_minutes` | int / NULL | Бесплатных минут в день (NULL = безлимит, 0 = нет бесплатных) | 10 |
| `subscription_tiers` | table | Уровни подписки с дневными лимитами и ценами | Pro (30 мин/день) |
| `welcome_bonus_minutes` | int | Бонусные минуты при регистрации | 60 |
| `welcome_bonus_days` | int / NULL | Срок действия бонуса (NULL = бессрочно) | 7 |
| `minute_packages` | table | Доступные пакеты минут с ценами | 50/100/500/5000 мин |

#### Преимущества подхода

- **Без деплоя**: изменение условий через БД, без пересборки контейнера
- **Гранулярность**: от общих условий до индивидуальных per-user
- **Версионность**: история изменений через цепочку условий с датами
- **A/B-тестирование**: разные условия для разных пользователей одновременно
- **Планирование**: будущие условия создаются заранее и вступают в силу автоматически
- **Полная гибкость**: дневной лимит можно обнулить (полностью платная модель), подписки и пакеты — независимые сущности

## 6. Финансовая модель на 6 месяцев

### 6.1. Предпосылки модели

**Ключевой инсайт**: при дневном лимите 10 мин Free-юзеры ограничены в потреблении. Пользователи, которым нужно больше (лекции, встречи, подкасты), покупают подписку или пакеты минут. Два канала дохода повышают общий ARPU.

**Фактические данные (февраль 2026)**:

| Тип файлов | Файлов | Минут | Расход (₽) | % расходов |
|-----------|--------|-------|-----------|-----------|
| Короткие (≤10 мин в день) | ~1 700 (76%) | ~2 100 | ~1 407 | ~17% |
| Сверх лимита 10 мин/день | ~530 (24%) | ~10 230 | ~6 855 | ~83% |
| **Итого** | **~2 230** | **~12 331** | **~8 262** | **100%** |

**С лимитом 10 мин/день**: расходы на Free-сегмент значительно снижаются. ~22% пользователей превышают лимит и являются целевой аудиторией для монетизации.

| Параметр | Значение | Обоснование |
|----------|---------|------------|
| Текущий MAU | 171 | Факт, февраль 2026 |
| MoM рост MAU | 0-10% | Реалистичная оценка (замедление с текущих 68%) |
| Churn-эффект от лимитов | -5% MAU | Единоразовый, лимит 10 мин/день покрывает 78% юзеров |
| Себестоимость 1 мин аудио | 0.67 ₽ | Whisper API rate |
| **Ср. расход на Free-юзера** | **~5 ₽/мес** | Оценка: ~7.5 мин/мес × 0.67₽ (лимит 10 мин/день) |
| **Ср. расход на Paid-юзера** | **~76 ₽/мес** | Факт: юзеры с длинными файлами |
| ARPU (пакеты, средневзвешенный) | ~150 ₽/мес | Средний пакет ~100 мин (175₽), не все тратят полностью |
| ARPU (подписка, средневзвешенный) | ~P_sub ₽/мес | Зависит от выбранного периода (неделя/месяц/год) |
| Telegram Stars комиссия | 30% | Apple/Google store fee |
| ЮKassa комиссия | 3.5% | Карты РФ + СБП |
| VPS (постоянные) | 672 ₽/мес | Факт |
| DeepSeek (переменные) | ~97 ₽/мес | Факт |

### 6.2. Модель дохода: два канала

**Канал 1: Пакеты минут**

```
Формулы (ЮKassa, комиссия 3.5%):
Расходы = (Free * 5₽) + (Paid * 76₽) + 769₽
Доход = Paid * ARPU_pkg * 0.965
Маржа на 1 Paid-юзера (ARPU 150₽) = 150 * 0.965 - 76 = +68.75₽ (прибылен!)
Маржа на 1 Paid-юзера (ARPU 175₽) = 175 * 0.965 - 76 = +92.88₽ (прибылен!)
```

**Канал 2: Подписки**

Подписка Pro (30 мин/день) добавляет предсказуемый recurring-доход. Подписчик также может покупать пакеты сверх подписки (файлы длиннее 30 мин). Для финмодели учитываем подписку как отдельный поток дохода.

> Примечание: Точные цены подписки определены в [требованиях к тарификации](tarification-requirements.md). В модели ниже расчёт приведён для пакетов минут как основного канала. Доход от подписок — **дополнительный** поток, улучшающий все сценарии.

### 6.3. Четыре сценария (пакеты минут, без учёта подписок)

**Формулы**:
```

Доход (ЮKassa) = Paid * 150 * 0.965 = Paid * 144.75
```

### 6.4. Анализ break-even

**Формула break-even (ЮKassa, пакеты)**:
```
Paid × 144.75 = (MAU - Paid) × 5 + Paid × 76 + 769
Paid × (144.75 + 5 - 76) = MAU × 5 + 769
Paid × 73.75 = MAU × 5 + 769
CR = (5 + 769/MAU) / 73.75
```

| MAU | BE CR (ARPU 150₽) | BE CR (ARPU 175₽) | BE CR (ARPU 200₽) |
|-----|-------------------|-------------------|-------------------|
| 163 | **13.4%** | **11.2%** | **9.6%** |
| 200 | **12.0%** | **10.0%** | **8.6%** |
| 300 | **10.3%** | **8.6%** | **7.4%** |
| 500 | **8.9%** | **7.4%** | **6.4%** |

> **Сравнение с v1** (модель с 20 мин/день): break-even CR снизился с 20-26% до 9-13% — значительное улучшение за счёт снижения расхода на Free-юзера (5₽ vs 9₽).

> **С учётом подписок**: break-even CR ещё ниже, т.к. ARPU подписчиков выше ARPU пакетных покупателей.

## Итоговые выводы и рекомендации

### 1. Модель «10 мин/день + подписки + пакеты минут»

- **Free**: 10 бесплатных минут в день, любая длительность файла, AI-кнопки без ограничений
- **Подписка Pro**: 30 мин/день, 3 периода (неделя/месяц/год) со скидками 33% и 53%
- **Пакеты минут**: 50/100/500/5 000 мин, бессрочные, цены 1.00–2.00₽/мин
- **Welcome-бонус**: 60 минут для новых пользователей (настраиваемый срок)
- **Грейс-период**: 60 бессрочных минут для существующих пользователей
- **Платёжные провайдеры**: Telegram Stars (мелкие покупки) + ЮKassa (основной канал)

### 2. Экономика модели (улучшение vs v1)

- Break-even CR снижен с 20-26% до **9-13%** — за счёт уменьшения расхода на Free-юзера
- При CR 15% модель **прибыльна с 1-го месяца** (даже без подписочного дохода)
- Подписки добавляют предсказуемый recurring-доход и дополнительно улучшают экономику
- Даже в наихудшем сценарии экономия ~8 000₽/мес vs без монетизации

### 3. Два рычага оптимизации

- **Дневной лимит**: настраивается в БД от 0 до ∞ — главный рычаг баланса retention/маржа
- **ARPU**: стимулирование покупки крупных пакетов (визуальное выделение «Популярный», upsell при исчерпании лимита)

### 4. Архитектура с запасом на будущее

- Произвольное количество уровней подписки (1, 2, 4 и т.д.)
- Индивидуальные условия для VIP-пользователей и акций
- Все условия с датами действия — автоматическое вступление в силу
- Дневной лимит может быть обнулён для перехода на полностью платную модель
- Подписки и пакеты — независимые сущности, работают параллельно
