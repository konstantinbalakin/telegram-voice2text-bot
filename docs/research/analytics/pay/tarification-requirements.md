# Система тарификации: требования к разработке

**Дата**: 1 марта 2026
**Статус**: Готово к планированию архитектуры
**Контекст**: [Полный аналитический документ](2026-02-28-monetization-model.md)

---

## Проблема

Бот работает в полностью бесплатном режиме. Расходы на OpenAI Whisper API составляют ~300₽/день и растут экспоненциально. Основная причина — небольшой процент пользователей (~14%) транскрибирует длинные аудио (лекции, встречи, видео), генерируя ~60% всех расходов. Себестоимость 1 минуты аудио — 0.67₽.

---

## Выбранная модель монетизации

**«Дневная квота + Подписки + Пакеты минут»**

Единый дневной лимит бесплатных минут для всех пользователей (10 мин/день). Сверх лимита — подписка или покупка пакетов. Подписочная модель запускается сразу с одним активным тарифом (30 мин/день). Архитектура позволяет в будущем полностью убрать бесплатные минуты и оставить только подписки и пакеты.

---

## Пользовательские требования

### 1. Дневной бесплатный лимит

- Каждый пользователь получает **10 бесплатных минут в день**
- Лимит обнуляется ежедневно в 00:00 MSK
- Никаких ограничений на длительность файла или количество транскрипций — только суммарные минуты в день
- AI-кнопки (Структурировать, Сделать красиво, О чём этот текст) — без ограничений (кнопки доступны только после транскрипции, отдельная отправка текста для обработки не поддерживается)
- Значение лимита настраивается в БД без деплоя
- **Архитектурное требование**: значение может быть установлено в 0 — это полностью отключает бесплатные минуты для всех пользователей, оставляя только подписки и пакеты

### 2. Индивидуальные условия для пользователей

- Возможность переопределить количество бесплатных минут в день для конкретного `user_id`
- Возможность переопределить любой другой параметр:
  - Дневной лимит (увеличить до 30/50 мин или сделать безлимитным)
  - Индивидуальные пакетные предложения
  - Индивидуальный welcome-бонус
  - Индивидуальные условия подписки
- Приоритет: **индивидуальные условия > общие условия (оферта)**
- Если индивидуальный параметр не задан (NULL) — берётся из общих условий

### 3. Подписочная модель

#### Активный тариф

На старте один активный уровень подписки:

| Параметр | Значение |
|----------|----------|
| Название | Pro (рабочее) |
| Дневной лимит | 30 мин/день |

#### Периоды и ценообразование

| Период | Цена | Скидка от недельного |
|--------|------|----------------------|
| Неделя | 69 рублей разово | — |
| Месяц | 199 рублей разово (199 рублей/мес) | **-33%** |
| Год | 1668 рублей разово (139 рублей/мес) | **-53%** |

**Логика скидок**:
- Недельный тариф, пересчитанный на 30 дней = 296 рублей
- Месячный пакет даёт скидку **33%** от стоимости недельного, пересчитанного на месяц
- Годовой пакет даёт скидку **53%** от стоимости недельного, пересчитанного на месяц

#### Взаимодействие подписки и бесплатного лимита

- Подписка **заменяет** бесплатный дневной лимит (не суммируется)
- Пользователь с Pro-подпиской получает **30 мин/день**, а не 10 (free) + 30 (подписка) = 40
- Бонусные минуты при покупке подписки **сохраняются** и продолжают действовать по обычным правилам списания

#### Отмена, продление и смена подписки

- **Возврат**: возврат средств за подписку **не предусмотрен** (стандартная практика подписочных сервисов)
- **Отмена**: при отмене подписка действует **до конца оплаченного периода**, затем не продлевается. Пользователь возвращается на бесплатный дневной лимит
- **Авто-продление**:
  - ЮKassa: автоматическое рекуррентное продление (с явным согласием пользователя при первой оплате)
  - Telegram Stars: не поддерживает рекуррентные платежи — пользователь продлевает вручную; бот отправляет напоминание за 3 дня до истечения
- **Upgrade** (повышение тарифа/периода): применяется **немедленно**, без пересчёта остатка старой подписки. Новая подписка стартует с текущей даты
- **Downgrade** (понижение): применяется **по истечении текущего оплаченного периода**. Пользователь оставляет заявку, при следующем продлении переключается на более низкий тариф
- В архитектуре предусмотреть поле `next_subscription_tier_id` — что будет при следующем продлении (покрывает и downgrade, и отмену)

#### Архитектурные требования к подпискам

- Возможность создавать **произвольное количество уровней** подписки (1, 2, 4 и т.д.)
- Каждому уровню задаётся: название, дневной лимит минут, описание, порядок отображения
- Доступные периоды: **неделя, месяц, год** (в будущем возможен квартал)
- Для каждой комбинации (уровень × период) задаётся отдельная цена
- Подписка автоматически продлевается или истекает по дате окончания
- Пользователь может иметь только одну активную подписку

### 4. Пакеты минут

- Если аудио превышает оставшийся дневной лимит — пользователь покупает пакет минут
- Пакеты **бессрочные** (не сгорают)
- Предварительная лестница тарифов:

| Пакет | Цена | ₽/мин | Скидка |
|-------|------|-------|--------|
| 50 мин | 100₽ | 2.90 | — |
| 100 мин | 175₽ | 1.75 | -13% |
| 500 мин | 750₽ | 1.5 | -25% |
| 5000 мин | 5000₽ | 1.00 | -50% |

- Система должна быть **гибкой**: возможность создавать пакеты на любое количество минут с любой ценой
- Каждому пакету задаётся: название, кол-во минут, цена, порядок отображения

### 5. Welcome-бонус для новых пользователей

- При первом запуске бота — **60 бонусных минут**
- Срок действия настраивается: 7 дней / бессрочно / другой период
- Бонус тратится **только** когда дневной лимит исчерпан
- Настройки бонуса (количество минут, срок) — в БД, без деплоя

### 6. Грейс-период для существующих пользователей

- При запуске монетизации все текущие пользователи получают **60 бессрочных бонусных минут** (одноразовый миграционный скрипт), которые будут оформлены как бонусные минуты без срока
- Если пользователь укладывается в дневной лимит — бонус не тратится
- **Уведомление**: при запуске монетизации отправляется broadcast-сообщение всем существующим пользователям с дружеским предупреждением об изменениях и начислении бонусных минут

### 7. Логика списания минут

#### Округление

- Длительность аудио округляется **до десятых долей минуты** (например, 2 мин 37 сек → 2.6 мин)
- Все балансы хранятся и отображаются с точностью до десятых

#### Приоритет списания

1. **Дневной лимит** (бесплатные минуты — общие или из подписки; подписка **заменяет** бесплатный лимит, не суммируется)
2. **Бонусные минуты** (welcome / грейс)
3. **Пакетные минуты** (купленные)

Минуты **размазываются** между источниками: одна транскрипция может частично списаться из дневного лимита, частично из бонуса и частично из пакета.

#### Обработка очереди

При одновременной отправке нескольких аудио — проверка лимита и списание выполняются **последовательно** (через Queue Manager). Каждое следующее аудио проверяется с учётом уже списанных минут из предыдущих.

#### Два сценария при отправке аудио

**А) Суммарно минут достаточно** (дневной остаток + бонус + пакет >= длительность аудио):
- Транскрипция выполняется **бесшовно**, без остановки
- После транскрипции — **обязательное уведомление**: «Дневной лимит закончился. Использовано X мин из пакета. Остаток: Y мин»

**Б) Суммарно минут недостаточно**:
- Транскрипция **не выполняется**
- Сообщение: «Недостаточно минут. Длительность: N мин. Доступно: X мин»
- Инлайн-кнопки покупки подходящих пакетов или подписки

**В) Уведомление о приближении к лимиту**:
- При использовании **80%+ дневного лимита** — предупреждение после транскрипции: «Осталось X мин из дневного лимита»
- Порог уведомления (80%) настраивается через ENV

### 8. Сроки действия условий

- Каждое условие (общее и индивидуальное) имеет `valid_from` и `valid_to` (nullable = бессрочно)
- Позволяет: менять оферту с конкретной даты, создавать временные акции, планировать будущие изменения
- Условия вступают в силу автоматически по дате

### 9. Пользовательский интерфейс

- **Обновлённое приветственное сообщение**: условия (10 мин/день бесплатно + 60 бонусных минут)
- **`/balance`**: остаток дневного лимита, бонусные минуты, пакетные минуты, статус подписки
- **`/subscribe`**: каталог подписок с ценами, скидками и кнопками оплаты
- **`/buy`**: каталог пакетов с ценами, скидками и кнопками покупки
- **`/help`**: условия использования, включая финансовые

### 10. Платёжные провайдеры

- **Telegram Stars** (Bot Payments API) — нативно, zero friction, комиссия ~30%
- **ЮKassa** (карты РФ, СБП) — комиссия 3.5%, основной канал по марже
- ЮKassa: формирование чеков в соответствии с 54-ФЗ (требование законодательства)

### 11. Метрики для отслеживания

При планировании архитектуры необходимо учесть, что новые таблицы (daily_usage, purchases, subscriptions и т.д.) должны позволять строить следующие метрики. Существующий аналитический скрипт `docs/research/analytics/generate_excel_report.py` должен быть расширен новыми листами для монетизационных метрик.

#### Conversion Metrics (еженедельно)

| Метрика | Целевое значение |
|---------|-----------------|
| % пользователей, превысивших дневной лимит | 10-20% |
| Free-to-Paid CR (купившие хотя бы 1 пакет или подписку) | 5-10% |
| Повторные покупки (% от покупателей) | >40% |
| ARPU (платящих) | 100-200 ₽/мес |

#### Retention Metrics (ежемесячно)

| Метрика | Целевое значение |
|---------|-----------------|
| Churn (MAU) до/после монетизации | <120% от baseline |
| DAU/MAU Ratio | >20% |

#### Financial Metrics (ежемесячно)

| Метрика | Целевое значение |
|---------|-----------------|
| Gross Margin (на платящих) | >50% |
| Расходы на Free-сегмент | Снижение vs текущее |
| LTV (paid user) | >300 ₽ |
| Расходы/MAU | Снижение тренда |

#### Operational Metrics (еженедельно)

| Метрика | Целевое значение |
|---------|-----------------|
| Среднее использование дневного лимита | 30-50% |
| Среднее время до первой покупки | <14 дней |
| Использование welcome-бонуса (% новых) | >30% |

## Ограничения

- Все настройки тарифов — **в БД**, не в ENV/config (в env файле, возможно, включение флага, что работаем с платным режимом)
- Изменение условий — **без деплоя**
- Бесплатные минуты в день — настраиваемый глобальный параметр, который можно обнулить для перехода на полностью платную модель
- Подписки и пакеты — две независимые сущности, могут работать параллельно или по отдельности
